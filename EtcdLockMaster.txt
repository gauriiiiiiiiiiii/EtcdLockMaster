==============================================================================
  WHAT THIS PROJECT DOES
==============================================================================

EtcdLockMaster is a distributed locking library. It solves this problem:

  Multiple processes on multiple machines need to coordinate access to a
  shared resource — a database, a file, an external API, a job queue.
  Only ONE process should be doing the work at a time. If two processes
  run simultaneously, you get data corruption, duplicate jobs, race conditions.

A distributed lock enforces this:
  - Process A acquires the lock — it is the only one allowed to proceed
  - Processes B, C, D wait (or time out)
  - Process A finishes and releases the lock
  - The next waiter acquires it and proceeds

Extra guarantee: if Process A CRASHES mid-work, the lock is automatically
freed after a TTL (time-to-live) expires. No manual cleanup. No zombie locks.

Real-world uses:
  - Run a database migration from exactly one service instance at a time
  - Ensure only one Kubernetes pod fires a scheduled cron job
  - Serialize writes to a third-party API that rejects concurrent requests
  - Leader election in a microservices cluster

==============================================================================
  WHAT ETCD IS — THEORY AND INTERNALS
==============================================================================

etcd is a distributed key-value store for coordination data: small,
critical state that multiple machines must agree on. It is the storage
backend of Kubernetes — every pod, secret, config, and service lives here.

Unlike Redis or Postgres (which can return stale data under failure),
etcd is STRONGLY CONSISTENT: every read from any node always returns
the latest confirmed write.

RAFT CONSENSUS ALGORITHM
--------------------------
etcd uses Raft to keep all nodes in sync:

  1. LEADER ELECTION
     Nodes elect one leader. All writes go through the leader.
     Followers replicate — they do not accept writes directly.

  2. WRITE COMMIT
     a. Leader appends to local WAL (write-ahead log)
     b. Sends to all followers
     c. Once MAJORITY acknowledge (quorum), write is committed
     d. Client receives success
     3-node: 2/3 must confirm. 5-node: 3/5 must confirm.

  3. LINEARIZABILITY
     After commit, every read from any node returns that value.
     No stale reads. No split-brain.

  4. FAULT TOLERANCE
     3-node survives 1 failure. 5-node survives 2.
     If majority is lost, cluster stops writes (safety over availability).

  5. WHY IT MATTERS FOR LOCKS
     Two Python processes on DIFFERENT etcd nodes still share the same
     lock state. Both nodes have the same data because every write was
     committed by a Raft quorum.

LEASES AND TTL
---------------
A lease is a server-managed time-limited token with a unique integer ID.
Keys written with a lease ID live and die with the lease.

  client.lease(30)         create lease, TTL = 30 seconds
  put(key, value, lease)   key bound to this lease
  lease.refresh()          reset the TTL countdown
  lease.revoke()           immediately delete lease and ALL its keys

If the lock holder crashes:
  - Nobody calls refresh() -> TTL reaches zero
  - etcd deletes all keys on that lease automatically
  - Watching clients receive DeleteEvent, can now compete for the lock
  - No external cleanup process needed. This is the crash-recovery mechanism.

ATOMIC TRANSACTIONS (COMPARE-AND-SWAP)
----------------------------------------
  client.transaction(
      compare=[condition],
      success=[op_if_true],
      failure=[op_if_false],
  )

Executes atomically on the server. No other client can interleave.

We use this for lock acquisition:
  IF   create_revision("/locks/resource") == 0    (key does not exist)
  THEN put("/locks/resource", owner_uuid, lease_id)
  ELSE do nothing

create_revision is the global cluster revision when the key was first
created. 0 means never written. Safer than checking for an empty value
because a client cannot fake absence by writing an empty string.

  got_it = True  -> we own the lock
  got_it = False -> someone else holds it

WATCHES (SERVER-SENT EVENTS)
------------------------------
A watch is a persistent subscription to a key. etcd pushes events
the instant a change happens — no polling needed.

  events, cancel = client.watch("/locks/resource")
  for ev in events:
      if isinstance(ev, DeleteEvent):
          # lock released

The events iterator blocks with zero CPU until etcd sends something.
DeleteEvent fires on key deletion (lease revoked or TTL expired).
PutEvent fires on writes — the code checks isinstance and ignores these.

Timeout: arm threading.Timer(wait_time, cancel). When it fires, cancel()
stops the iterator, the for loop exits, the function returns False.


==============================================================================
  HOW THE LOCK WORKS — FULL STEP BY STEP
==============================================================================

ACQUIRING A LOCK
-----------------
acquire_lock("resource", ttl=30, timeout=10):

  1. Record start time. Begin retry loop.
  2. if elapsed >= timeout: raise TimeoutError
  3. client.lease(ttl) — fresh lease each attempt (crash-safety timer)
  4. uuid.uuid4() — fresh owner ID each attempt (122 bits of entropy)
  5. Atomic transaction: IF create_revision(key)==0 THEN put(key, uuid, lease)
  6a. got_it=True -> return (owner, lease)
  6b. got_it=False:
        a. safe_revoke(lease) — discard the unused lease immediately
        b. remaining = timeout - elapsed  (or None if no timeout)
        c. _wait_for_deletion(key, remaining) — block until key deleted
        d. Timeout? raise TimeoutError
        e. DeleteEvent? loop back to step 2

  Fresh lease + fresh UUID every iteration. Never reuse state from a
  failed attempt — a reused lease could already be close to expiry.

RELEASING A LOCK
-----------------
release_lock("resource", owner, lease):

  1. GET current value from etcd.
  2. val is None -> key already gone (TTL expired). Log warning. Return.
  3. val == owner UUID -> safe_revoke(lease):
       Revoke -> etcd deletes key -> all watchers get DeleteEvent -> wake up
  4. val != owner UUID -> raise RuntimeError (ownership mismatch, refuse)

AUTO-RENEWAL
-------------
With auto_renew=True, a background thread keeps the lease alive:

  interval = self.ttl / 2.0
  while not self._stop_event.wait(interval):
      self.lease.refresh()

Uses _stop_event.wait(interval) NOT time.sleep(interval).
wait() wakes immediately when the event is set — instant clean shutdown.
sleep() would make __exit__ wait up to TTL/2 more seconds before stopping.

Thread lifecycle:
  __enter__: create Event, start daemon thread
  __exit__:  _stop_event.set() -> thread wakes and exits
             _renew_thread.join() -> wait for clean exit
             then release_lock()

Why TTL/2: even if one refresh is delayed by nearly TTL/2 seconds (GC
pause, network hiccup), the next fires before expiry. At TTL-1s interval,
any single delay breaks the lock.

Why daemon=True: if the process is SIGKILL'd, a regular thread would
keep the process alive forever. Daemon thread dies with the process.
Lease stops refreshing, etcd cleans up via TTL.

RE-ENTRANCY
------------
_count tracks nesting depth:

  __enter__, _count==0  -> acquire_lock, _count=1
  __enter__, _count>0   -> _count += 1, return self (no network call)
  __exit__              -> _count -= 1
                          if _count==0: release_lock

Without _count, nested "with lock:" calls acquire_lock() again.
Key exists with OUR UUID -> create_revision != 0 -> transaction FAILS.
We block waiting for ourselves. Permanent deadlock.

__exit__ does NOT suppress exceptions from the with block. It catches
only exceptions from release_lock itself. The original exception from
inside the block propagates normally after the lock is released.


==============================================================================
  EVERY FUNCTION — LINE BY LINE
==============================================================================

MODULE SETUP
-------------
  Logging: DEBUG to console + distributed_lock.log (gitignored).

  client = etcd3.client(host="127.0.0.1", port=2381)
  Module-level. Functions read client at CALL TIME, not import time.
  Monkey-patchable:
      import distributed_lock as dl
      dl.client = etcd3.client(port=2379)
  is_locked property also uses module-level client — also patchable.

safe_revoke(lease)
-------------------
  try: lease.revoke()
  except: swallow "lease not found", log everything else

  "Lease not found" happens when TTL expired naturally or we are cleaning
  up a lease from a failed acquire. Both are expected. Key is already gone.

_wait_for_deletion(key, wait_time)
------------------------------------
  Fast path (wait_time <= 0): GET key, return val is None. No blocking.

  events, cancel = client.watch(key)       <- WATCH FIRST (race-safe)
  val, _ = client.get(key)                 <- THEN check
  if val is None: cancel(); return True    <- race-window deletion caught

  If the key was deleted between our failed transaction and this call,
  the DeleteEvent is already queued in the watch stream. Caught either
  by the GET returning None, or by the event arriving in the loop.
  If we checked first and watched second, that deletion would be missed.

  Infinite wait (wait_time=None): iterate events loop, no timer.
  Timed wait: threading.Timer(wait_time, cancel) arms a cancellation.
  finally: timer.cancel() — always runs, prevents timer leak.

acquire_lock(resource, ttl, timeout)
--------------------------------------
  key = f"/locks/{resource}"  — all locks under /locks/ prefix.
  Bulk cleanup: delete_prefix("/locks/").

  remaining = (timeout - elapsed) if timeout is not None else None
  Passes None directly. Old buggy code did `remaining or 0` which
  turned None into 0 — instant return — spurious TimeoutError.

release_lock(resource, owner, lease)
--------------------------------------
  Ownership check before release: safety guard against releasing a lock
  we no longer own (our lease may have expired, someone else grabbed it).

  safe_revoke(lease) not client.delete(key) because:
    1. Revoke atomically handles ALL keys on that lease
    2. Works if lease already expired (safe_revoke handles not-found)
    3. Watchers receive DeleteEvent immediately

DistributedLock class
----------------------
  __init__: resource, ttl, timeout, auto_renew stored.
            owner=None, lease=None, _count=0, thread stuff=None.

  __enter__: re-entrancy fast path if _count>0.
             acquire_lock, _count=1.
             Start daemon renew thread if auto_renew.

  __exit__:  _count -= 1, return if still nested.
             Signal thread (_stop_event.set), join it.
             release_lock in try/except (log error, don't re-raise).

  is_locked: live etcd GET every call, not cached.
             Use for safety checks, not tight loops.


==============================================================================
  RACE CONDITIONS — EVERY BUG AND FIX
==============================================================================

RACE 1 — Key deleted between transaction failure and watch setup
-----------------------------------------------------------------
Without fix:
  t0  B fails transaction (A holds lock)
  t1  A releases -> key deleted
  t2  B starts watch
  t3  Nothing fires, key already gone — B times out unnecessarily

Fix — watch BEFORE existence check:
  t0  B fails transaction
  t1  B starts watch                     <- ACTIVE
  t2  A releases -> DeleteEvent queued in B's watch stream
  t3  B does GET -> None -> cancel, return True immediately
      OR: DeleteEvent arrives in events loop if GET races ahead

Both orderings of t2/t3 are handled. Race window closed.

RACE 2 — timeout=None treated as timeout=0
--------------------------------------------
Buggy code: _wait_for_deletion(key, remaining or 0)
When timeout=None: remaining=None. None or 0 = 0.
Fast path triggers, returns immediately. TimeoutError raised even
though user asked to wait forever.

Fix: pass remaining directly. None = "block forever" branch.

RACE 3 — Monkey-patched client not restored between tests
----------------------------------------------------------
test_concurrent_acquire_different_clients patches distributed_lock.client
in threads. Afterwards module-level client points at wrong port.
etcd_cleanup fixture (autouse) uses stale client -> fails -> all later
tests break.

Fix: _original_client = distributed_lock.client
     try: ... threads ...
     finally: distributed_lock.client = _original_client

==============================================================================
  DOCKER — WHERE AND HOW IT IS USED
==============================================================================

Docker is an ALTERNATIVE to running the Windows etcd binaries manually.
It is not required — the project works fine with a single standalone node.

WHERE IT IS USED
-----------------
docker-compose.yml defines a 3-node etcd cluster:

  etcd1  — client port 2379, peer port 2380
  etcd2  — client port 2381, peer port 2383   <- app connects here
  etcd3  — client port 2382, peer port 2384

All three containers join one Raft cluster via --initial-cluster.
Docker's internal DNS resolves container names (etcd1, etcd2, etcd3)
to container IPs so peers can reach each other.

HOW TO START IT
----------------
  docker compose up -d       starts all 3 containers in background
  docker compose down        stops and removes containers
  docker compose logs -f     tail all container logs

WHY PORT 2381
--------------
The app (distributed_lock.py) connects to host:2381 by default.
In docker-compose.yml, etcd2 maps host:2381 -> container:2379.
So the app connects to etcd2 in the cluster. This is the "app node"
in the original 3-node design.

WHEN YOU NEED DOCKER (vs standalone binary)
---------------------------------------------
Standalone binary (single node):
  - Works for all development and testing
  - 11 tests pass, 1 skipped (test_concurrent_acquire_different_clients)
  - Simplest — one terminal, one command

Docker 3-node cluster:
  - All 12 tests pass (port 2379 becomes reachable)
  - Simulates a real production-grade distributed cluster
  - Tests cluster-wide lock exclusivity across different nodes
  - Required only if you want to run all 12 tests or simulate HA


==============================================================================
  API REFERENCE
==============================================================================

CONTEXT MANAGER (recommended)
-------------------------------
  from distributed_lock import DistributedLock

  try:
      with DistributedLock("my-resource", ttl=30, timeout=10, auto_renew=True) as lock:
          print("Lock held by", lock.owner)
          print("Still locked?", lock.is_locked)
          # ... critical section ...
  except TimeoutError:
      print("Could not acquire lock in time")

LOW-LEVEL FUNCTIONS
--------------------
  from distributed_lock import acquire_lock, release_lock

  owner, lease = acquire_lock("my-resource", ttl=30, timeout=10)
  # ... critical section ...
  release_lock("my-resource", owner, lease)

  Use low-level when you need to acquire in one place and release in
  another, or when you need direct access to the lease object.

PARAMETERS
-----------
  resource   str    (required)  Lock name — any unique string
                                e.g. "db-migration", "cron-job", "api-write"

  ttl        int    default=30  Lease lifetime in seconds.
                                If the holder crashes, lock auto-frees after TTL.
                                Set higher for slower tasks. Keep low for fast ones.

  timeout    float  default=None  Max seconds to wait for the lock.
                                  None = wait forever.
                                  Raises TimeoutError if exceeded.

  auto_renew bool   default=False  Start a background thread that refreshes
                                   the lease every TTL/2 seconds.
                                   Use for tasks that take longer than TTL.

PROPERTIES
-----------
  lock.owner      str   UUID string identifying this lock holder.
                        Stored in etcd as the key's value.

  lock.is_locked  bool  Live etcd GET on every call (not cached).
                        True if key exists AND value matches our UUID.
                        False if lease expired, lock released, or never acquired.
                        Use for safety checks, not tight loops.


==============================================================================
  TROUBLESHOOTING
==============================================================================

'.' is not recognized
  You ran a PowerShell command in CMD.
  Use the CMD version: no .\ prefix, no backtick line continuations.
  Put the full command on one line.

bind: Only one usage of each socket address
  A port is already in use (usually 2380 or 2382 — the peer port).
  Change --listen-peer-urls and --initial-cluster to a free port,
  e.g. use 2389 instead of 2382.

db file is flocked by another process
  An old etcd process is still running and holding the data file lock.
  Kill it first (Ctrl+C or Task Manager), then:
    rmdir /s /q etcd-data
  Then restart etcd with a fresh --data-dir.

server has been already initialized
  The data directory has stale cluster state from a previous run with
  different configuration. Same fix:
    rmdir /s /q etcd-data

grpc._channel._InactiveRpcError / etcd connection failed / request timed out
  Python cannot reach etcd, or etcd is running but stuck (no leader elected).
  The full reset procedure:
    1. taskkill /F /IM etcd.exe
    2. rmdir /s /q etcd-data
    3. Start etcd fresh (see README Step 2 command)
    4. Wait for: "serving client traffic insecurely, address 127.0.0.1:2381"
  Cause of "stuck" etcd: stale data directory that remembers a different
  cluster configuration from a previous run. Deleting etcd-data forces a
  clean single-node initialization.

11 passed, 1 skipped (not 12 passed)
  Expected behavior with a single standalone node. The skipped test
  (test_concurrent_acquire_different_clients) needs port 2379 to be
  reachable. Run docker compose up -d to get all 12 passing.

==============================================================================
  DESIGN DECISIONS — EVERY TRADEOFF
==============================================================================

1. LEASE-BASED CLEANUP vs HEARTBEAT MONITOR
   Postgres + cron job: extra service to deploy, clock skew risk.
   Etcd leases: server-enforced TTL, no external dependency.

2. ATOMIC CAS vs READ-THEN-WRITE
   Read-then-write: two clients both read "empty", both write, both win.
   Atomic transaction: one server-side op, cannot be split.

3. WATCH vs POLLING
   Polling: wastes CPU, 0-100ms latency, unnecessary etcd load.
   Watch: notification within milliseconds, zero CPU while waiting.

4. WATCH-BEFORE-CHECK vs CHECK-BEFORE-WATCH
   Check-before-watch: deletion in the gap = missed = unnecessary timeout.
   Watch-before-check: closes the race window. Was a bug in original code.

5. TTL/2 REFRESH INTERVAL vs TTL-epsilon
   TTL-epsilon (e.g. TTL-1s): single delayed refresh = expired lock.
   TTL/2: full TTL of buffer. Two consecutive missed refreshes still safe.

6. _stop_event.wait() vs time.sleep() IN RENEW LOOP
   sleep(): thread waits up to TTL/2 after stop signal. Slow shutdown.
   wait(): wakes immediately when event is set. Instant clean shutdown.

7. DAEMON THREAD vs REGULAR THREAD
   Regular: keeps process alive if __exit__ never called (SIGKILL).
   Daemon: dies with process. Lease stops, etcd cleans up via TTL.

8. MODULE-LEVEL CLIENT vs DEPENDENCY INJECTION
   Injection: verbose for users, cleaner in large systems.
   Module-level: one-line import and use, overridable via monkey-patch.

9. DUAL API: LOW-LEVEL + CONTEXT MANAGER
   Low-level: for fine-grained control, acquire/release in different places.
   Context manager: release guaranteed even on exception, by Python itself.
