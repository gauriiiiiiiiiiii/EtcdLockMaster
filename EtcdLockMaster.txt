==============================================================================
  WHAT THIS PROJECT DOES
==============================================================================

EtcdLockMaster is a distributed locking library. It solves this problem:

  Multiple processes on multiple machines need to coordinate access to a
  shared resource — a database, a file, an external API, a job queue.
  Only ONE process should be doing the work at a time. If two try to run
  simultaneously, data corruption, duplicate jobs, or race conditions occur.

A distributed lock is the solution:
  - Process A acquires the lock → it is the only one allowed to proceed
  - Processes B, C, D wait (or time out)
  - Process A finishes and releases the lock
  - The next waiter acquires it and proceeds

Extra guarantee: if Process A CRASHES mid-work, the lock is automatically
freed after a TTL (time-to-live) expires. No manual cleanup. No zombie locks.

Real-world uses:
  - Run a database migration from exactly one service instance at a time
  - Ensure only one Kubernetes pod fires a scheduled cron job
  - Serialize writes to a third-party API that rejects concurrent requests
  - Leader election in a microservices cluster

==============================================================================
  WHAT ETCD IS — INTERNALS AND THEORY
==============================================================================

etcd is a distributed key-value store built for coordination data:
small amounts of critical state that multiple machines must agree on.
It is the storage backend of Kubernetes — every pod, secret, config,
and service definition is stored in etcd.

Unlike Redis or Postgres (which can serve stale reads under failure),
etcd is STRONGLY CONSISTENT: every read from any node always returns
the latest confirmed write.

RAFT CONSENSUS ALGORITHM
--------------------------
etcd uses the Raft algorithm to keep all nodes synchronized:

  1. LEADER ELECTION
     At startup, nodes elect a leader. All writes go through the leader.
     Followers only replicate data — they do not accept writes directly.

  2. WRITE COMMIT
     When a client writes a key:
       a. Leader receives the write and appends it to its local WAL log.
       b. Leader sends the write to all followers.
       c. Once a MAJORITY of nodes acknowledge (quorum), write is committed.
       d. The client receives a success response.
     In a 3-node cluster: 2 out of 3 must confirm.

  3. LINEARIZABILITY
     Once a write is committed, every subsequent read from any node
     returns that value. There is no "stale read" problem.

  4. FAULT TOLERANCE
     A 3-node cluster survives 1 node failure.
     A 5-node cluster survives 2 node failures.
     If a majority is lost, the cluster stops accepting writes.

  5. WHY THIS MATTERS FOR LOCKS
     Two Python processes can connect to DIFFERENT etcd nodes and still
     share the same lock state. Both nodes agree on who holds the lock
     because every write was committed by a Raft quorum.

LEASES AND TTL
---------------
A lease is a server-managed time-limited token with a unique integer ID.
When you write a key and attach a lease ID, that key lives and dies with
the lease.

  client.lease(30)         creates a lease with 30-second TTL
  put(key, value, lease)   key is bound to this lease
  lease.refresh()          resets the 30-second countdown
  lease.revoke()           immediately deletes the lease and all its keys

If the process holding the lock crashes:
  - Nobody calls refresh() → TTL countdown reaches zero
  - etcd deletes all keys attached to the lease automatically
  - Clients watching the key receive a DeleteEvent
  - They can now compete for the lock again

This is the crash-recovery mechanism. No external cleanup job needed.

ATOMIC TRANSACTIONS (COMPARE-AND-SWAP)
----------------------------------------
etcd transactions have three parts: compare, success, failure.

  client.transaction(
      compare=[condition],
      success=[operation_if_true],
      failure=[operation_if_false],
  )

The entire thing executes atomically on the server.

We use this for lock acquisition:
  IF   create_revision("/locks/resource") == 0   (key does not exist)
  THEN put("/locks/resource", owner_uuid, lease_id)
  ELSE (do nothing)

`create_revision` is the cluster revision at which a key was first created.
If the key has never been written, this value is 0.
Using this (instead of checking if value is empty) is safer because a
client cannot fake "key doesn't exist" by writing an empty string.

  got_it = True  → we own the lock
  got_it = False → key already exists, someone else holds it

WATCHES (SERVER-SENT EVENTS)
------------------------------
A watch is a persistent subscription on a key or key prefix. etcd pushes
events to watchers the instant a change happens on the server.

  events, cancel = client.watch("/locks/resource")
  for ev in events:
      if isinstance(ev, DeleteEvent):
          # lock was released

The events iterator blocks with zero CPU usage until etcd sends something.
No polling, no sleep loop, no wasted syscalls.

Timeout is implemented by arming a threading.Timer that calls cancel()
after wait_time seconds. cancel() stops the events iterator.


==============================================================================
  HOW THE LOCK WORKS — COMPLETE STEP BY STEP
==============================================================================

ACQUIRING A LOCK
-----------------
acquire_lock("resource", ttl=30, timeout=10):

  1. Record start time. Begin retry loop.

  2. Check if timeout has elapsed:
       if elapsed >= timeout: raise TimeoutError

  3. Create a new lease with the given TTL.
     Crash-safety timer. If we crash after writing the key, the key
     disappears automatically after TTL seconds.

  4. Generate a UUID4 as the owner ID.
     122 bits of random entropy. Unique across all machines and time.

  5. Run the atomic transaction:
       IF create_revision(key) == 0: put(key, uuid, lease_id)
       ELSE: nothing

  6a. Transaction succeeded:
        Return (owner_uuid, lease). Caller owns the lock.

  6b. Transaction failed (someone else holds the key):
        a. Immediately revoke the lease we just created.
        b. Calculate remaining = timeout - elapsed (or None)
        c. Call _wait_for_deletion(key, remaining)
        d. If timeout: raise TimeoutError
        e. If DeleteEvent received: loop back to step 2 and retry

RELEASING A LOCK
-----------------
release_lock("resource", owner, lease):

  1. GET current value from etcd.
  2. If None: key already gone (TTL expired). Log warning. Return.
  3. If value == our owner UUID: revoke the lease.
     Revoke → etcd deletes key → watchers receive DeleteEvent → wake up.
  4. If value != our UUID: raise RuntimeError (ownership mismatch).

AUTO-RENEWAL
-------------
With auto_renew=True, a background daemon thread refreshes the lease:

  interval = ttl / 2.0
  while not _stop_event.wait(interval):
      lease.refresh()

The thread starts in __enter__ after acquisition.
In __exit__: _stop_event.set() → _renew_thread.join() → release_lock().

Why TTL/2: gives a full TTL of buffer. A single missed refresh (network
hiccup, GC pause) does not expire the lock.

Why daemon=True: if the process is killed before __exit__ runs, a daemon
thread dies automatically. The lease stops refreshing. etcd cleans up
via TTL expiry. Correct crash behavior.

RE-ENTRANCY
------------
DistributedLock keeps an internal integer _count:

  __enter__, _count == 0:  acquire from etcd, _count = 1
  __enter__, _count > 0:   just increment _count (no network call)
  __exit__:                decrement _count
                           if _count == 0: release from etcd

Without _count, the second "with lock:" would call acquire_lock() again.
The key exists with our own UUID. The transaction would FAIL. We would
block waiting for ourselves. Permanent deadlock.


==============================================================================
  THE CODE — EVERY FUNCTION EXPLAINED
==============================================================================

MODULE SETUP
-------------
  logging configured at DEBUG level to both console and distributed_lock.log.

  client = etcd3.client(host="127.0.0.1", port=2381)

  Module-level. All functions reference `client` at call time (not import
  time). Tests can monkey-patch it:
      import distributed_lock as dl
      dl.client = etcd3.client(port=2379)
  All subsequent calls in that scope use the patched client.

safe_revoke(lease)
-------------------
  try: lease.revoke()
  except: swallow "lease not found", log everything else

  Revoking an already-expired lease throws "lease not found". This is
  expected and harmless — the key is already gone. We swallow it silently.
  All other exceptions are still logged.

_wait_for_deletion(key, wait_time)
------------------------------------
  Fast path: if wait_time <= 0, just GET and return val is None.

  events, cancel = client.watch(key)    WATCH STARTS FIRST

  val, _ = client.get(key)              EXISTENCE CHECK AFTER
  if val is None: cancel(); return True

  Race-safe ordering: watch must start before the existence check.
  If the key was deleted between the failed transaction and this call,
  the watch stream already has a queued DeleteEvent, or GET returns None.
  Either way the deletion is caught.

  If wait_time is None: block forever in for ev in events loop.
  With timeout: arm threading.Timer(wait_time, cancel). Timer fires →
  cancel() stops the events iterator → for loop exits → return False.

acquire_lock(resource, ttl, timeout)
--------------------------------------
  key = f"/locks/{resource}"
  All locks under /locks/ prefix for easy bulk cleanup.

  Retry loop: fresh lease + fresh UUID each iteration (never reuse state
  from a failed attempt). Atomic transaction. On failure: revoke lease,
  wait, retry.

release_lock(resource, owner, lease)
--------------------------------------
  GET key. Verify ownership. Revoke lease (not direct delete) because:
    1. Revoke atomically deletes ALL keys on that lease.
    2. Revoke is a no-op if lease already expired (safe_revoke handles it).
    3. Watchers receive DeleteEvent immediately on revoke.

DistributedLock class
----------------------
  __init__: stores params, _count=0, owner/lease/thread=None.

  __enter__:
    Re-entrancy fast path if _count > 0.
    acquire_lock(), _count=1.
    Start renew thread if auto_renew=True.
    Return self.

  __exit__:
    Decrement _count. Return if still nested.
    Signal renew thread to stop, join it.
    Call release_lock(). Log error on exception but don't re-raise.
    (Exception from inside the with block propagates normally.)

  is_locked property:
    Live etcd GET on every call. val is not None and val == self.owner.
    Use to detect if lease expired mid-operation without auto_renew.


==============================================================================
  RACE CONDITIONS — EVERY BUG AND ITS FIX
==============================================================================

RACE 1 — Key deleted between transaction failure and watch setup
-----------------------------------------------------------------
Without fix:
  t0  B's transaction fails (A holds lock)
  t1  A releases → key deleted
  t2  B calls client.watch(key)
  t3  Nothing fires → B times out unnecessarily

Fix — watch BEFORE existence check:
  t0  B's transaction fails
  t1  B calls client.watch(key)     watch is ACTIVE
  t2  A releases → DeleteEvent queued in B's watch stream
  t3  B calls GET(key) → None → cancel, return True immediately
      (OR: DeleteEvent arrives in events loop if GET races ahead)

RACE 2 — timeout=None treated as timeout=0
--------------------------------------------
Original buggy code:
    saw_delete = _wait_for_deletion(key, remaining or 0)

When timeout=None: remaining=None. `None or 0` = 0.
_wait_for_deletion(key, 0) returns immediately.
Result: TimeoutError even though user said "wait forever".

Fix: pass remaining directly. Handle None as a separate "block forever"
branch inside _wait_for_deletion.

RACE 3 — Monkey-patched client not restored between tests
----------------------------------------------------------
test_concurrent_acquire_different_clients patches distributed_lock.client
in threads. After threads finish, module-level client points at wrong port.
etcd_cleanup fixture fails → all subsequent tests break.

Fix: save original client, restore in try/finally.
    _original = distributed_lock.client
    try: ... run threads ...
    finally: distributed_lock.client = _original

==============================================================================
  DEPENDENCIES — WHY EACH VERSION IS PINNED
==============================================================================

etcd3==0.12.0
  Python client for etcd v3 API. Provides lease(), transaction(),
  watch(), get(), delete_prefix(). Not actively maintained — pinned to
  avoid breaking changes in future versions.

grpcio==1.62.3
  etcd communicates over gRPC (HTTP/2-based RPC framework).
  etcd3 uses grpcio as its transport. Pinned to match what etcd3 0.12.0
  was tested against.

protobuf==3.20.3
  Serialization format for gRPC messages. etcd3 ships pre-compiled Python
  stubs generated by the protobuf 3.x code generator. In protobuf 4.x,
  several internal APIs used by these stubs were removed.
  Result: importing etcd3 with protobuf 4.x raises AttributeError.
  THIS MUST STAY AT 3.x. Do not upgrade without regenerating all stubs.

pytest==8.3.5
  Test runner. pytest.ini suppresses DeprecationWarning from grpcio.


==============================================================================
  DESIGN DECISIONS — EVERY TRADEOFF
==============================================================================

1. LEASE-BASED CLEANUP vs EXTERNAL HEARTBEAT MONITOR
   Alternative: store locks in Postgres, cron job deletes stale rows.
   Problems: extra service to deploy/monitor, clock skew risk.
   Etcd leases: TTL enforced server-side, no external dependency.

2. ATOMIC CAS TRANSACTION vs READ-THEN-WRITE
   Read-then-write is not atomic. Two clients can both read "empty" and
   both write — both think they won. Mutual exclusion is broken.
   The etcd transaction is ONE server-side atomic op. Unbreakable.

3. WATCH-BASED WAITING vs POLLING
   Polling wastes CPU, creates etcd load, adds latency.
   A watch delivers notification within milliseconds, zero CPU while waiting.

4. WATCH-BEFORE-CHECK vs CHECK-BEFORE-WATCH
   See Race 1 above. Check-before-watch has a race window.
   Watch-before-check closes it. Non-obvious. Was a bug in original code.

5. TTL/2 REFRESH INTERVAL vs TTL-epsilon
   TTL-epsilon leaves almost no margin. One delayed refresh breaks lock.
   TTL/2 gives a full TTL of buffer. Two consecutive misses still safe.

6. DAEMON THREAD vs REGULAR THREAD
   Regular thread keeps process alive if __exit__ never called (SIGKILL).
   Daemon thread dies with the process. Lease stops refreshing.
   etcd cleans up via TTL. Correct crash behavior.

7. MODULE-LEVEL CLIENT vs DEPENDENCY INJECTION
   Dependency injection is cleaner in large systems but verbose for users.
   Module-level client enables one-line import and use. Still overridable
   via monkey-patch for multi-client test scenarios.

8. DUAL API: LOW-LEVEL + CONTEXT MANAGER
   Low-level: for acquire-in-one-function, release-in-another patterns.
   Context manager: for clean with blocks — release guaranteed even on
   exception, because __exit__ is always called by Python's with machinery.
